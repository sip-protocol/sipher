## Summary

Backfill direct unit-level test coverage for the **29 agent tools** that the 2026-04-18 audit flagged as missing per-tool tests: **8 user-facing tools** under `packages/agent/src/tools/`, **all 14 SENTINEL tools** under `packages/agent/src/sentinel/tools/` (currently lumped under two umbrella files), and **7 HERALD tools** under `packages/agent/src/herald/tools/`. Tests follow a uniform contract — heavy service-layer mocks, real Zod parsing, real tool body, no I/O — modelled on the depth set by Phase 4 service-integration tests (~10-15+ tests per tool, scaled to tool complexity). Phase 5 ships across **three PRs by subsystem** (user-facing → SENTINEL → HERALD), with one shared spec (this doc) and three implementation plans generated by the writing-plans skill at the start of each PR. Estimated ~250-400 new tests added; zero source changes.

## Goals

1. Add **one direct test file per tool** for each of the 29 tools, named to match the source file exactly.
2. Apply a **uniform six-row test sheet** per tool (see Architecture → Test sheet contract below). Range: 5-8 tests for simple read tools, 15-20+ for action tools with branches.
3. **Migrate** every meaningful assertion from the two SENTINEL umbrella files (`read.test.ts`, `action.test.ts`) and any equivalent HERALD-side overlap into per-tool homes; delete umbrellas only after migration verified passing.
4. **Zero regression** in existing root suite (35 files, 555 tests), agent suite (75 files, 938 tests), app suite (45 tests).
5. **Zero source-code changes** to the 29 tools — Phase 5 is test-only. If a test reveals a real bug, file a follow-up issue and pause; do not fix in the same PR.
6. **Three PRs**, one per subsystem, in order: PR-1 user-facing → PR-2 SENTINEL → PR-3 HERALD.
7. **Single shared spec** (this doc) covers all 3 PRs. **Three plans** (one per PR) generated by writing-plans at PR start.

## Architecture

### Mock depth — heavy service-layer mocks

Each tool is a thin wrapper that parses inputs, calls one or more services, and formats the result. The test contract is therefore narrow: **does the tool wire its inputs to the right service call and return the right shape?** Phase 4 already validated the services themselves.

Mock depth per subsystem:

| Subsystem | Mocked at boundary | Real |
|-----------|-------------------|------|
| User-facing | `src/services/*` modules — `chainTransferBuilder`, `privateSwapBuilder`, `transactionBuilder`, `cspl`, `helius-provider`, `solana`, etc. | Zod parsing, tool body, type guards |
| SENTINEL | `src/agents/sipher/db.ts` query helpers, `src/sentinel/tx-builder.ts`, Pi agent calls (`@mariozechner/pi-ai`) | Zod parsing, tool body |
| HERALD | `src/herald/x-client.ts` (the only X-API surface) | Zod parsing, tool body |

Real Zod parsing is non-negotiable — input validation is the tool's primary contract and tests must exercise it.

A tool test should run in **<50ms**. The whole 29-tool Phase 5 suite combined should add no more than a few seconds to CI wall time.

### Test sheet contract — six rows per tool

Every tool gets the same checklist, scaled to the tool's branching:

1. **Happy path.** Normal inputs, normal output. 1-2 tests; if the tool has variants (e.g., token vs. SOL), one happy path per variant.
2. **Schema rejects bad inputs.** Each Zod constraint that can fail produces one test asserting the rejection error shape. Use `it.each` for parameterized validation cases.
3. **Each internal branch covered.** Every `if` / `switch` / token-vs-native / chain-dispatch / has-cspl / has-cache path gets at least one test. For tools that dispatch over `chain` (e.g., `balance`, `send`, `claim`), this means one test per supported chain at minimum.
4. **Service errors surface clean.** When the mocked service throws, the tool surfaces the right error shape (custom error class, error message, HTTP status if applicable). One test per distinguishable error kind.
5. **Output shape locked.** Required fields present, types correct, no surprise extras. Use a small assertion helper or per-test object spread to capture the contract.
6. **Spy assertion on service call.** The mock service is a `vi.fn()` — tests assert `mock.mock.calls[0]` shows the exact args the tool passed (including any defaulted or transformed values). This row is the single most important assertion for tool tests because mocking the service means the spy is the only proof the tool wired things correctly.

Range: read tools get 5-8 tests, action tools with branches get 15-20+. Concrete per-tool counts will be set in each PR's plan.

### Umbrella migration policy (PR-2 and PR-3)

For PR-2's two SENTINEL umbrellas (`read.test.ts`, `action.test.ts`):

1. **Read each umbrella line-by-line before writing new files.** Catalog every assertion: which tool it covers, what it asserts.
2. **Each assertion finds a per-tool home.** A test about `add-to-blacklist`'s blacklist-write behavior moves into `add-to-blacklist.test.ts`. A test about Pi agent prompt construction (if shared across action tools) moves into `cross-tool.test.ts`.
3. **Cross-tool assertions get their own file.** If an umbrella holds genuine cross-tool invariants (e.g., "all read tools return `{ ok: true, data }` shape"), they land in `tests/sentinel/tools/cross-tool.test.ts`. Use this sparingly — most assertions belong to a specific tool.
4. **Delete the umbrella only after.** Once every assertion has a new home AND the new tests pass at full Phase 5 depth (the six rows), delete the umbrella file in the same commit that completes its last migrated assertion. Don't carry both forward.
5. **Verify nothing was lost.** The PR description must include a checklist confirming each umbrella `it(...)` block was migrated or explicitly judged redundant (with rationale).

PR-3 has no SENTINEL-style umbrellas — `read-mentions.test.ts` and `post-tweet.test.ts` are already per-tool files for tools we're NOT touching in Phase 5. They stay as-is. PR-3 only adds 7 new files.

**Vitest hoisting note (Phase 4 lesson):** `vi.mock(path, importedFactoryCall())` fails because `vi.mock` is hoisted above imports — the imported factory is in TDZ at hoist evaluation. Workaround: inline the factory inside each test file, calling helper functions only inside test bodies (not in `vi.mock` arguments). Fixture files export factory functions used inside tests / `beforeEach` / `it` bodies, not as `vi.mock` arguments.

## Risks and alternatives

**Doing all 29 tools in one mega-PR.** Rejected — Phase 4 was 3 files in one PR and was already heavy; 29 tools across three subsystems with three different mock surfaces would create reviewer fatigue and merge-conflict risk. Three PRs by subsystem is the natural cut.

**Splitting by risk tier (read vs action) instead of by subsystem.** Rejected — would mix subsystems in each PR, requiring three different mock setups per PR. Subsystem split keeps each PR's mock surface uniform.

**Keeping the SENTINEL umbrellas as-is and adding per-tool files alongside.** Rejected — duplicate coverage, two-places-to-update for any behavior change, future devs uncertain which file to touch. Migration once is cheaper than duplication forever.

**Deleting umbrellas without migration ("trust new tests cover it").** Rejected — Phase 4 lesson reinforced that "obviously redundant" old assertions sometimes encode hidden invariants. Migration is one extra read-and-place pass per umbrella; cheap insurance.

**Testing tools at integration depth (real services + mocked network), matching Phase 4's pattern.** Rejected — Phase 4 already validated the services. Re-paying integration setup cost on tool tests would add slow tests without adding signal. Heavy service mocks keep tool tests focused on tool wiring.

**Running tools through their AgentCore registration (`tool-adapter`, `tools.ts` umbrella) rather than calling them directly.** Rejected — that's what existing `tests/tools.test.ts` and `tests/tool-adapter.test.ts` do. Phase 5's value is direct unit-level proof of each tool's contract, not yet-another-layer of agent integration.

**One spec per PR (three specs total).** Rejected — the architectural decisions (mock depth, test sheet contract, umbrella policy, fixture strategy) are uniform across all three subsystems. One spec captures the shared contract; three plans execute it.

**Adding tests for the 14 already-covered user-facing tools "while we're here."** Rejected — out of audit scope, would more than double Phase 5 size, and the existing tests presumably pass and weren't flagged. Separate ticket if any prove thin.

## Implementation notes

- **Existing pattern reference for new test files:** the closest analogue is `packages/agent/tests/swap.test.ts` (existing direct tool test for the `swap` tool) and `packages/agent/tests/sweep.test.ts`. Match their import style and `vi.mock` usage. Phase 4's `tests/private-swap-builder.test.ts` is also a good reference for spy assertion style.
- **Mock the source path the tool imports from.** Each tool's `import { foo } from './services/x.js'` translates to `vi.mock('./services/x.js', ...)` inside the test file. Resolve relative to the test file's location.
- **Spy assertions:** use `expect(mockFn).toHaveBeenCalledWith(...)` for the simple case; use `expect(mockFn.mock.calls[0]).toEqual([...])` for finer control over partial-match logic.
- **`it.each` for parameterized validation tests** — each Zod constraint that can fail becomes one row in the table. Keeps schema-validation tests dense and readable.
- **Run order:** within each PR, write tests in the order `happy path → schema → branches → errors → output → spy`. This mirrors the test sheet rows and makes review predictable.
- **No source changes.** If a test reveals a real bug, file a follow-up GitHub issue with the failing assertion and link it from the PR description. Do not fix in the Phase 5 PR.
- **CI verification per PR:** `pnpm --filter @sipher/agent test -- --run` must pass. Root suite (`pnpm test -- --run`) and app suite (`pnpm --filter @sipher/app test -- --run`) must remain green.
- **Spec + plan + impl on the same branch (Phase 4 lesson #4):** PR-1's branch is `feat/phase-5-user-tool-tests` and holds this spec, the PR-1 plan, and the PR-1 implementation commits. PR-2's branch (`feat/phase-5-sentinel-tool-tests`) is created from main *after* PR-1 merges; it holds only the PR-2 plan + impl. Same for PR-3.
- **Estimated total wall time:** ~3-4 sessions (one per PR). Each PR follows the brainstorm-already-done → writing-plans → subagent-driven-development → request-review pattern.
